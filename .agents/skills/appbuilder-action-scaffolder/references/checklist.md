# App Builder Action Scaffolder Checklist

- Scope and expected outcome are explicitly stated — Verify the request names the action, files, and intended result; How: inspect the prompt or plan before editing; Pass: one concrete deliverable and any out-of-scope work are written down; Common failure: vague requests hide whether to scaffold, modify, or deploy.
- Assumptions and dependencies are listed — Verify required inputs, secrets, CLIs, packages, and services are called out; How: inspect the plan and referenced files for env vars, `aio`, and package expectations; Pass: missing prerequisites are named before work starts; Common failure: implementation assumes credentials or tooling that were never confirmed.
- Security and data handling implications are reviewed — Verify the change will not log secrets or expose sensitive payloads; How: inspect code and examples for `console.log`, hardcoded keys, and unredacted tokens; Pass: secrets stay in config and logs only show safe fields; Common failure: debug output leaks auth headers, tokens, or customer data.
- Runtime limits, latency, and cost impacts are checked — Verify the action fits Adobe Runtime limits and avoids unnecessary work; How: inspect for large deps, long loops, serial network calls, or unbounded retries; Pass: execution stays bounded and any slow external call is justified; Common failure: oversized bundles or retry loops push runs over time or memory limits.
- Tests or reproducible verification steps are included — Verify there is a command or exact manual flow to prove the change; How: look for `npm test`, `aio app dev` (or `aio app run` for actions using State SDK, Files SDK, or sequences), curl examples, or documented steps; Pass: another agent can rerun the check without guessing; Common failure: the handoff says "should work" without a command, payload, or expected output. Note: `aio app dev` runs actions locally but State SDK (`@adobe/aio-lib-state`) and Files SDK (`@adobe/aio-lib-files`) require a Runtime environment — use `aio app run` when testing actions that depend on those SDKs.
- Rollback or mitigation path is documented — Verify the response explains how to undo or contain the change if deployment fails; How: inspect the handoff for revert, disable, or fallback steps; Pass: a concrete backout or mitigation path is written; Common failure: recovery guidance starts only after failure in a live environment.
- Handoff summary contains next steps and ownership — Verify the final note says what changed, what remains, and who should act next; How: inspect the closing summary or task note; Pass: the next action and owner are explicit; Common failure: the handoff stops at "done" with no follow-up guidance.
- `app.config.yaml` keeps action configuration under `application.runtimeManifest` — Verify runtime entries live under `application.runtimeManifest` or the extension-specific config the template expects; How: inspect `app.config.yaml` and confirm the nested key path; Pass: action definitions sit under the application block the CLI reads; Common failure: runtime config is placed at the wrong level, so deployments silently skip it.
- No root-level `runtimeManifest` key exists in `app.config.yaml` — Verify the file does not define `runtimeManifest:` at column 1 because the CLI ignores it; How: run `grep -n '^runtimeManifest:' app.config.yaml` or inspect top-level keys; Pass: no root-level match is found; Common failure: a copied manifest block looks valid in YAML but never deploys.
- Memory setting uses a pre-warmed container size (256 / 512 / 1024 MB) to avoid cold starts — Verify the action's `limits.memory` value in `app.config.yaml` is one of the three pre-warmed sizes; How: run `grep -A2 'limits:' app.config.yaml` and check memory values; Pass: memory is 256, 512, or 1024 (or omitted, which defaults to 256); Common failure: a non-standard value like 384 or 2048 is set, bypassing the pre-warmed container pool and causing cold starts on every invocation, which adds noticeable latency.
- Web action logging is enabled when debugging — Verify that debugging instructions include the `x-ow-extra-logging: on` header for web actions; How: check curl examples and test commands for the header; Pass: any debugging or verification step that invokes a web action over HTTP includes the header so logs are actually captured; Common failure: a developer follows the verification steps, sees no logs for a successful call, and assumes the action did not run or the logger is broken.
- Two-layer auth is intentional and both layers are accounted for — Verify that actions using `require-adobe-auth: true` (manifest annotation) AND code-level `Authorization` header checks (`params.__ow_headers.authorization`) have both layers deliberately configured; How: run `grep -rn 'require-adobe-auth' app.config.yaml src/*/ext.config.yaml` and `grep -rn '__ow_headers.*authorization\|Authorization.*Bearer' actions/ src/*/actions/`; Pass: each auth-protected action either uses only one layer or documents why both are needed, and local dev instructions explain how to bypass both for testing; Common failure: a user toggles one auth layer but not the other, causing silent 401 errors that persist across redeploy cycles.
